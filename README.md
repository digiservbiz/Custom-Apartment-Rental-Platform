# Custom Apartment Rental Platform

## 1. Project Overview

This project is a full-stack web application designed as a custom-built apartment rental platform. It connects apartment owners and renters, providing a comprehensive suite of tools for managing listings, handling bookings, processing payments, and facilitating communication.

The platform is built with a modern MERN-like stack (MongoDB, Express, React, Node.js) and integrates several third-party services to provide real-world functionality, including Stripe for payments, Twilio for WhatsApp messaging, and SendGrid for email notifications.

### Core Features

*   **User Roles & Authentication:** Supports multiple user roles (Renter, Owner, Agent, Admin) with a robust authentication system using JWTs. Users can register via email/password or social providers (Google, Facebook).
*   **KYC Verification:** A manual KYC (Know Your Customer) system requires administrators to approve Owners and Agents before they can list properties, ensuring a trusted community.
*   **Apartment Listings:** Approved users can create, update, and manage detailed apartment listings with photos, descriptions, pricing, and more.
*   **Real-time Availability Checks:** A unique feature allowing renters to request an apartment's availability via a button that triggers a WhatsApp message to the owner. The owner's reply directly updates the apartment's status in the database.
*   **Secure Booking & Payments:** A complete booking flow integrated with Stripe for secure payment processing. The system handles pending bookings, payment confirmation via webhooks, and prevents double-bookings.
*   **Automated Notifications:** Sends automated booking confirmations to both renters and owners via email (SendGrid) and WhatsApp (Twilio).
*   **Review & Rating System:** Renters can leave reviews for apartments after a confirmed stay. These reviews are subject to admin moderation before being displayed publicly.
*   **Comprehensive Admin Panel:** A dedicated UI for administrators to manage users, KYC requests, review moderation, and platform settings like the commission fee.

## 2. Architecture

The application is architected as a modern monorepo with a separate frontend and backend.

### Backend

The backend is a Node.js application built with the Express framework. It follows a standard, modular structure to separate concerns, making it easier to maintain and scale.

*   **Framework:** Express.js
*   **Database:** MongoDB with Mongoose for object data modeling (ODM).
*   **Authentication:** JSON Web Tokens (JWT) for session management. Passport.js is used to handle OAuth for Google and Facebook social logins.
*   **Structure:**
    *   `src/config`: For database and Passport.js configuration.
    *   `src/controllers`: Contains the business logic for each route.
    *   `src/models`: Defines the Mongoose schemas for Users, Apartments, Bookings, etc.
    *   `src/routes`: Defines the API endpoints.
    *   `src/middleware`: Contains custom middleware for authentication (`authMiddleware`) and error handling.
    *   `src/services`: Contains logic for interacting with external APIs (Stripe, SendGrid, Twilio).
    *   `src/utils`: Utility functions and classes.

### Frontend

The frontend is a single-page application (SPA) built with React.

*   **Framework:** React (bootstrapped with Create React App).
*   **Routing:** `react-router-dom` is used for all client-side routing.
*   **State Management:** React's Context API (`AuthContext`) is used for managing global authentication state.
*   **API Communication:** The `axios` library is used for making all HTTP requests to the backend API.
*   **Structure:**
    *   `client/src/pages`: Contains top-level components that correspond to a page/route.
    *   `client/src/components`: Contains reusable components used across multiple pages (e.g., forms, alerts, spinners).
    *   `client/src/context`: Holds the React context providers for global state.
    *   `client/src/i18n.js`: Configuration for internationalization.

## 3. Getting Started

Follow these instructions to get a copy of the project up and running on your local machine for development and testing purposes.

### Prerequisites

*   Node.js and npm
*   MongoDB (local installation or a cloud service like MongoDB Atlas)
*   Accounts for the following services:
    *   Google (for OAuth)
    *   Facebook (for OAuth)
    *   Stripe (for payments)
    *   SendGrid (for emails)
    *   Twilio (for WhatsApp)

### Installation

1.  **Clone the repository:**
    ```sh
    git clone <repository-url>
    cd <repository-directory>
    ```

2.  **Install backend dependencies:**
    ```sh
    npm install
    ```

3.  **Install frontend dependencies:**
    ```sh
    npm install --prefix client
    ```

4.  **Set up environment variables:**
    *   Create a `.env` file in the root directory by copying the example: `cp .env.example .env`.
    *   Create a `.env` file in the `client` directory as well. This is for frontend-specific variables.
    *   Populate these files with your credentials as described in the table below.

## 5. API Endpoints

Below is a detailed documentation of the available API endpoints.

---

### 5.1 Authentication (`/api/v1/auth`)

#### `POST /register`
*   **Description:** Registers a new user.
*   **Access:** Public
*   **Request Body:**
    ```json
    {
      "name": "John Doe",
      "email": "john.doe@example.com",
      "password": "password123",
      "role": "renter"
    }
    ```
*   **Success Response (200):**
    ```json
    {
      "success": true,
      "token": "ey..."
    }
    ```

#### `POST /login`
*   **Description:** Logs in an existing user.
*   **Access:** Public
*   **Request Body:**
    ```json
    {
      "email": "john.doe@example.com",
      "password": "password123"
    }
    ```
*   **Success Response (200):**
    ```json
    {
      "success": true,
      "token": "ey..."
    }
    ```

#### `GET /me`
*   **Description:** Gets the details of the currently logged-in user.
*   **Access:** Private (requires valid token)
*   **Success Response (200):**
    ```json
    {
      "success": true,
      "data": { "_id": "...", "name": "John Doe", "email": "...", ... }
    }
    ```

#### `PUT /updatedetails`
*   **Description:** Updates the name and email of the logged-in user.
*   **Access:** Private
*   **Request Body:**
    ```json
    {
      "name": "Johnathan Doe",
      "email": "john.doe.new@example.com"
    }
    ```

#### `PUT /updatepassword`
*   **Description:** Updates the password of the logged-in user.
*   **Access:** Private
*   **Request Body:**
    ```json
    {
      "currentPassword": "password123",
      "newPassword": "newpassword456"
    }
    ```

---

### 5.2 Apartments (`/api/v1/apartments`)

#### `GET /`
*   **Description:** Gets a list of all apartments with filtering and pagination.
*   **Access:** Public

#### `POST /`
*   **Description:** Creates a new apartment.
*   **Access:** Private (Admin, Owner, Agent)

#### `GET /:id`
*   **Description:** Gets a single apartment by its ID.
*   **Access:** Public

#### `PUT /:id`
*   **Description:** Updates an apartment.
*   **Access:** Private (Owner of apartment, Admin)

#### `POST /:id/check-availability`
*   **Description:** Triggers a WhatsApp message to the owner to check for availability.
*   **Access:** Private (Renter, Admin)

---

### 5.3 Bookings (`/api/v1/bookings`)

#### `POST /`
*   **Description:** Creates a new booking with a `Pending` status. This is the first step in the booking process.
*   **Access:** Private (Renter)
*   **Request Body:**
    ```json
    {
      "apartment": "apartmentId",
      "checkInDate": "2025-12-20",
      "checkOutDate": "2025-12-25",
      "totalPrice": 500
    }
    ```

---

### 5.4 Payments (`/api/v1/payments`)

#### `POST /create-payment-intent`
*   **Description:** Creates a Stripe Payment Intent for a given booking ID.
*   **Access:** Private (Renter)
*   **Request Body:**
    ```json
    {
      "bookingId": "bookingId"
    }
    ```
*   **Success Response (200):**
    ```json
    {
      "success": true,
      "clientSecret": "pi_..."
    }
    ```

#### `POST /stripe-webhook`
*   **Description:** Webhook endpoint for Stripe to send events. This is called by Stripe's servers, not by the client.
*   **Access:** Public

---

### 5.5 Users (`/api/v1/users`) - Admin Only

All endpoints under `/api/v1/users` require admin privileges.

#### `GET /`
*   **Description:** Gets a list of all users.

#### `PUT /:id/updatestatus`

*   **Request Body:**

## 6. How It Works: Key Feature Guides

### 6.1 Booking & Payment Flow (Stripe)

The payment and booking confirmation process is designed to be robust and secure, using a combination of frontend and backend actions.

1.  **Booking Creation:** A logged-in renter selects their dates on the `BookingPage`. When they click "Confirm", the frontend first sends a request to `POST /api/v1/bookings`. The backend checks for date conflicts and, if available, creates a new booking record with a `Pending` status.
2.  **Payment Intent:** After successfully creating the pending booking, the frontend immediately makes a second request to `POST /api/v1/payments/create-payment-intent`, sending the new `bookingId`.
3.  **Client Secret:** The backend uses the Stripe SDK to create a **Payment Intent** and returns its `client_secret` to the frontend.
4.  **Stripe Elements:** The frontend uses the `client_secret` to render a secure payment form using **Stripe Elements**. The user's card details are sent directly to Stripe and never touch our server.
5.  **Payment Confirmation:** The user submits the payment form. Stripe processes the payment and, upon success, redirects the user to a confirmation page.
6.  **Webhook Fulfillment:** Independently, Stripe's servers send a `payment_intent.succeeded` event to our public webhook endpoint (`POST /api/v1/payments/stripe-webhook`). Our backend verifies this event came from Stripe, finds the booking using the metadata, and updates its status to `Confirmed`. This is the single source of truth for a successful booking.
7.  **Notifications:** Once the booking is confirmed via the webhook, the backend triggers the email and WhatsApp notification services to inform the renter and owner.

### 6.2 WhatsApp Availability Check (Twilio)

This feature allows renters to get near-real-time availability confirmation from property owners.

1.  **Renter Request:** A logged-in renter on an apartment details page clicks the "Check Availability" button.
2.  **Backend Trigger:** The frontend sends a request to `POST /api/v1/apartments/:id/check-availability`.
3.  **Message to Owner:** The backend finds the apartment's owner, sets the apartment's status to `Pending Confirmation`, and uses Twilio to send a pre-formatted WhatsApp message to the owner's phone number (e.g., "A renter is interested... Reply 'Yes [ApartmentID]' or 'No [ApartmentID]'").
4.  **Owner Reply:** The owner replies directly to the message on WhatsApp.
5.  **Webhook Processing:** Twilio sends this reply to our public webhook endpoint (`POST /api/v1/whatsapp/webhook`).
6.  **Status Update:** The backend webhook handler parses the message. If it's a valid "Yes" reply and the sender's number matches the owner's, it updates the apartment's status to `Available`.

## 7. Running Tests

This project includes a backend testing suite using Jest and Supertest.

### Running the Test Suite

To run the tests, execute the following command from the root directory:

```sh
npm test
```

The testing framework uses an in-memory MongoDB server, so it runs in a completely isolated environment and will not affect your development database.

### How to Write New Tests

When adding a new feature, please add a corresponding test file in the `/tests` directory.

1.  **Create a new test file:** e.g., `tests/newfeature.test.js`.
2.  **Follow the existing structure:** Use `describe` to group related tests and `it` for individual test cases. Use `supertest` to make requests to the app.
3.  **Database Seeding:** The test setup will automatically connect to and clear the database before each test. You can create any necessary mock data (users, apartments, etc.) within a `beforeEach` block or at the start of your `it` block.
